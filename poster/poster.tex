\begin{posterblock}{Motivación y objetivo}
La seguridad de RSA depende de que los primos usados para construir cada módulo
$N=p\cdot q$ sean impredecibles y no se reutilicen. Si dos módulos comparten un
factor primo, basta calcular $\gcd(N_i,N_j)$ para comprometer ambas claves.

Este escenario es especialmente relevante en certificados digitales, porque la
clave pública RSA forma parte del certificado y puede obtenerse de forma
abierta. Los Registros de Transparencia de Certificados (\textit{Certificate
Transparency logs}) proporcionan, por tanto, una fuente masiva de claves reales
emitidas en Internet sobre la que es posible realizar auditorías criptográficas.

Este problema ya fue estudiado en la tesis \textit{Finding shared RSA factors in
the Certificate Transparency logs}, donde se analizan certificados reales
extraídos de CT logs mediante técnicas \textit{batch GCD} clásicas. Ese trabajo
demuestra el interés práctico de auditar los CT logs, pero también evidencia el
elevado coste computacional del análisis masivo.

Como mejora algorítmica, el artículo de Pelofske propone \textit{Binary Tree
Batch GCD}, un método más eficiente que evita parte del coste del enfoque
clásico basado en \textit{remainder tree}.

Este TFM conecta ambas líneas: toma como objetivo aplicado la búsqueda de
factores RSA compartidos en CT logs, evalúa el algoritmo \textit{Binary Tree
Batch GCD} frente al método clásico y desarrolla una implementación
C++17/GMP/OpenMP preparada para ejecución masiva en Altamira.
\end{posterblock}

\begin{posterblock}{Algoritmo e implementación}
El enfoque ingenuo requiere $\mathcal{O}(M^2)$ cálculos GCD para $M$ módulos
RSA. El \textit{batch GCD} clásico evita esta comparación par a par mediante un
\textit{product tree} y un \textit{remainder tree}, pero esta segunda estructura
tiene un coste elevado.

\textit{Binary Tree Batch GCD} elimina el \textit{remainder tree} completo:
durante el árbol de productos calcula GCD entre los hijos de cada nodo, agrega
los GCD no triviales en una variable $B$ y finalmente comprueba
$\gcd(N_i,B)$. Esta estrategia es más eficiente cuando los factores compartidos
son pocos respecto al número total de claves, como se espera en CT logs reales.

Las versiones Python de Pelofske se usan como referencia, pero su sobrecarga en
enteros grandes limita el escalado. Por ello se implementa el algoritmo en
C++17 con GMP y OpenMP, mejorando la aritmética multiprecisión, el control de
memoria y la paralelización.

La validación combina dos fases: primero experimentos sintéticos en local, con
módulos RSA de 1024 y 2048 bits y distintos números de claves débiles; después,
ejecución en Altamira para reproducir el benchmark con los tamaños originales
del artículo y atacar el conjunto real de certificados de CT logs.
\end{posterblock}

\begin{posterblock}{Resultados preliminares}
El procedimiento experimental se planteó de forma incremental. Primero se
desarrolló una implementación propia de \textit{Binary Tree Batch GCD} en
C++/GMP, con el objetivo de disponer de una versión más eficiente que las
implementaciones Python y adecuada para una ejecución posterior sobre datos
reales.

A continuación se generaron datos sintéticos con módulos RSA controlados y un
número conocido de claves débiles. Esta fase permitió comprobar que el algoritmo
detectaba correctamente los factores compartidos y estudiar cómo variaba el
tiempo de ejecución al aumentar el número de módulos.

\begin{center}
\includegraphics[width=0.94\postercolwidth]{assets/escalabilidad_batch_gcd.pdf}
\end{center}

Una vez validada la implementación propia, se utilizó el repositorio asociado al
artículo de Pelofske para reproducir una comparación más próxima al experimento
original. El repositorio aportaba las implementaciones Python de
\textit{Remainder Tree Batch GCD} y \textit{Binary Tree Batch GCD}, además de
parte de los primos usados para generar los módulos sintéticos.

Para integrarlo en el TFM fue necesario adaptar el código original: corregir
incompatibilidades con versiones actuales de Python y NumPy, crear
\textit{wrappers} de ejecución, normalizar la entrada de módulos y generar los
primos adicionales necesarios para evaluar también módulos RSA de 2048 bits.

\begin{center}
\includegraphics[width=0.94\postercolwidth]{assets/tiempo_pelofske.pdf}
\end{center}

Con ello se compararon los tres algoritmos sobre los mismos datos de entrada:
el método clásico en Python, el algoritmo de Pelofske en Python y la
implementación propia en C++/GMP, usando módulos RSA de 1024 y 2048 bits y
distintos números de claves débiles.
\end{posterblock}

\begin{posterblock}{Aceleración observada}
En promedio, \textit{Binary Tree} Python fue aproximadamente $6.1\times$ más
rápido que \textit{Remainder Tree} Python. La implementación C++/GMP fue
aproximadamente $116.5\times$ más rápida que \textit{Remainder Tree} Python y
$18.7\times$ más rápida que \textit{Binary Tree} Python.

El caso más exigente de la parrilla local corresponde a módulos RSA-2048,
$N=10000$ y \texttt{WEAK}=1000: el método clásico tardó 866.819 s, la versión
Python del algoritmo de Pelofske 152.803 s y la implementación C++/GMP 5.851 s.

La ejecución local sobre 601 módulos únicos no detectó factores compartidos y

sobre la seguridad de los CT logs: su función es validar que el flujo completo
funciona sobre datos reales.

El conjunto local es pequeño frente a los estudios previos y frente al objetivo
final en Altamira. Su utilidad es comprobar que los módulos reales llegan al
binario C++ en el formato correcto y que la deduplicación previa evita trabajo
redundante.

\begin{posterblock}{Ejecución pendiente en Altamira}
El resultado final del póster se sustituirá por la campaña en Altamira:

\begin{itemize}
  \item benchmark completo con las $N$ originales del artículo;
  \item ataque distribuido sobre el conjunto completo de certificados de CT
  logs;
  \item registro de tiempos, memoria pico, módulos RSA válidos, módulos únicos
  y factores compartidos encontrados.
\end{itemize}

El diseño permite lanzar trabajos por bloques con un parámetro $N$ modificable,
manteniendo el ataque global correcto mediante una comparación de cada bloque
contra el producto global de todos los módulos.

Esta separación permite explotar varios nodos sin convertir el análisis en una
comparación parcial por bloques: si dos claves comparten un primo aunque estén
en trabajos distintos, el método distribuido sigue pudiendo detectarlo.
\end{posterblock}

\begin{posterblock}{Estado actual}
\begin{itemize}
  \item Benchmark local validado frente a las implementaciones Python originales.
  \item Implementación C++/GMP preparada para ejecución HPC.
  \item Pipeline real validado: extracción, filtrado, deduplicación y análisis.
  \item Pendiente: sustituir resultados preliminares por la salida completa de
  Altamira.
\end{itemize}

La versión actual del póster usa resultados locales como marcador técnico. La
estructura está preparada para reemplazar la tabla y las cifras por los CSV de
Altamira cuando finalicen las ejecuciones.
\end{posterblock}

\begin{posterblock}{Conclusiones provisionales}
\begin{itemize}
  \item \textit{Binary Tree Batch GCD} reduce el coste práctico frente al método
  clásico basado en \textit{remainder tree}.
  \item La implementación C++17/GMP mejora de forma clara a las versiones
  Python del repositorio original.
  \item El flujo completo de extracción, normalización y análisis se ha validado
  sobre certificados reales.
  \item Altamira permitirá reemplazar los resultados provisionales por una
  evaluación a escala real.
\end{itemize}

El trabajo ya valida metodología e implementación. La contribución final será
cuantificar el comportamiento a escala y, si aparecen, reportar los factores RSA
compartidos encontrados en el conjunto completo de CT logs.
\end{posterblock}
